# 抽离Vue组合式函数时,什么时候要加上watch
## 总结

根据一个响应式数据的变化，去执行一段副作用（Side Effect）代码的时候

### 场景一：数据是异步获取的，且你需要给 `ref` 输入框赋初始值（最常见）

- **现象：** 页面刚打开时，数据（来自网络请求、或者还没加载完的本地存储）是 `undefined`。
    
- **为什么写 `computed` / `ref` 不行：** `ref(computed.value.title)` 是一次性的，数据到了它也收不到。
    
- **必须写 `watch` 的理由：** 你必须让一个“监听器”守在那里，一旦异步数据终于从后端（或缓存）传过来了，`watch` 立刻醒过来，把新值赋给表单的 `title.value`。
    
- **vue组件的文件可以不写的原因:** 因为vue底层ref变量和上面的html组件之间响应,如果下面的数值改变那么他就会
    

### 场景二：路由参数（如 ID）发生变化，但组件被复用了

假设你在浏览一个备忘录详情页（`/memo/1`），页面上有一个“相关推荐”，点击后跳转到另一个备忘录（`/memo/2`）。

- **现象：** 因为使用的是同一个 `.vue` 组件，Vue 为了性能不会销毁重建这个组件，导致 `<script setup>` 里面的代码**不会重新执行**。此时你直接拿到的 `id` 变了，但表单里的内容依然停留在老页面。
    
- **必须写 `watch` 的理由：** 必须监听 `route.params.id`。只要 ID 变了，就在 `watch` 内部重新去查一次本地存储或调用一次接口，把新数据刷到页面上。
    

### 场景三：需要把数据持久化同步到外部（如 LocalStorage、日志、埋点）

当你希望只要页面上的某个变量一变，就自动触发一个外部动作。

- **例子：** 自动保存功能。用户在输入框里每打一个字，你就想把它存到浏览器的 `localStorage` 里，或者发给后端保存。
    
- **必须写 `watch` 的理由：** Vue 模版只能帮你把数据渲染到屏幕上，它没办法帮你自动执行 `localStorage.setItem()`。你必须写一个 `watch` 盯着输入框变量，只要它变了，就执行保存代码。
    

### 场景四：联动修改（A 变了，B 也要跟着主动变，但 B 还要能被用户修改）

- **例子：** “城市选择器”。当用户把“省份”从 `广东省` 改成了 `北京市`，那么“城市”输入框应该立刻被清空或者自动变成 `北京市`。
    
- **为什么不能用 `computed`：** 这里的“城市”需要能被用户手动点击修改。如果是 `computed`，它是只读的，用户没办法在界面上自由选择。
    
- **必须写 `watch` 的理由：** 监听“省份”，只要省份一变，在 `watch` 的回调函数里写一句 `city.value = ''`，主动把城市清空。



总结
- **如果你只想“加工”数据，得到一个新结果（只读展示）** 👉 用 **`computed`**。
    
    - _例子：根据长宽计算面积、根据列表过滤出未完成的项、把名字转成大写。_
        
- **如果你要在数据变了以后，“做一件具体的事情”（去执行某段逻辑）** 👉 用 **`watch`**。
    
    - _例子：数据变了去发请求、数据变了去打印日志、数据变了去跳转路由、数据变了去给另一个 `ref` 赋值。_
